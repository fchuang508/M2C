## 会议内容与决议

### 自由讨论 1：共创营HTML宣传页迭代

- **事实**<br>1. 服务器地址为 https://xz502.xz17.tw/，后端为 Synology NAS（型号大致为Synology同类设备）。<br>2. 代码仓库需要账号与访问权限作为基础条件。<br>3. 可用的 token 来源包括 GPT 订阅、Cloudflare 订阅、Perplexity 订阅；R 与 GPT 订阅可用 Cortex。<br>4. token 与服务均为国外来源，但认为使用上可能不构成问题。<br>5. 可由现有条件再提供一个 API 以对接对应能力，并支持对接 VIP。

- **观点**<br>1. 提供“duck 壳”、直接 hosting、或加域名取决于更好做的方案。<br>2. 对国外 token/服务的使用应当没有显著障碍。

- **建议**<br>1. 可先放一个“duck 壳”搭建简单版本用于迭代。<br>2. 也可以直接采用 hosting 方案。<br>3. 可先加上网域名，比较后选择更易落地的方式。<br>4. 代码库部分需要再讨论如何更好地组织与交付访问。<br>5. 可进一步制作并提供 API，把 token/能力封装后供团队直接使用。


### 自由讨论 2：服务器连接密钥确认

- **事实**<br>1. 需要为服务器端提供相应的安排/配置。

- **判断**<br>1. 当前连接仍可能取决于服务器端是否已完成必要配置。

- **建议**<br>1. 进一步梳理服务器端应如何配置/安排，以便更好地完成连接。


### 自由讨论 3：代码部署与VPN连接

- **事实**<br>1. 代码部署不必一定要最后来做；也可由AI直接将代码推送到目标环境<br>2. 代码仓库在尚未准备开源前可放在任意主体处<br>3. 开VPN后才能建立连接（需通过VPN）<br>4. 如需开源，GitHub需要额外准备

- **判断**<br>1. 在集体处托管代码仓库更合适<br>2. 在某些情况下部署推送可能并不需要立即使用

- **观点**<br>1. 代码仓库放在集体名下更符合合适的组织方式<br>2. 开VPN是连接所必需的前提

- **建议**<br>1. 如尚未开源，可先将仓库放在集体处以便管理与协作<br>2. 需要连接时先开VPN再进行部署/访问


### 自由讨论 4：国内GitHub与宣传素材确认

- **事实**<br>1. 需确认国内GitHub相关内容<br>2. 除作品外，media部分是否还包含其他视频、是否拍照片合影需确认<br>3. 照片是否需要放入宣传需确认且照片要放的

- **建议**<br>1. 确认国内GitHub部分的具体内容与范围<br>2. 明确media部分除作品外是否补充其他视频与照片合影<br>3. 确定宣传物料中照片的使用方式与投放安排


### 自由讨论 5：GitHub合作与私密群组方案

- **事实**<br>1. 使用 GitHub 作为当前合作平台；发言者不确定是否需要再增加 VPN。<br>2. 提到亚马扣与“Get TGITEE”相关的扣费/访问场景，具体机制不明。<br>3. GitHub 私密群组（private group）被认为可能可用来实现“放在内地”的方案，尚未验证。<br>4. 视频存档可通过 notes/nurses 功能保存，发言者会进一步查看。<br>5. 发言者计划评估 NAS 作为 hosting/存储承载视频存档的可行性。

- **判断**<br>1. 再增加 VPN 可能带来更大阻力或影响现有流程，但目前不确定。<br>2. 将方案放在内地可能更符合当前倾向。<br>3. 从 GitHub 体系走可能更利于梳理合作方式与流程，并沉淀稳定做法。

- **观点**<br>1. 应优先选择能支持合作方式与流程沉淀的方案。<br>2. 可以通过把 iMeeting 需要做的事项整理出来并报价，形成更稳定的合作落地方式。<br>3. 在涉及访问与存储方案时，需要先研究验证可行性再决定。

- **建议**<br>1. 研究并尝试开设 GitHub 私密群组，用其承载合作/资料共享。<br>2. 在决定是否增加 VPN 前，评估其阻力与对现有合作流程的影响。<br>3. 准备 NAS/hosting 方案的选项，作为视频存档的备选承载方式。<br>4. 将 iMeeting 需要完成的事项整理成可报价清单，用于后续从 GitHub 内部推进合作。
